Here is a clean, interview-friendly documentation for **503, 504, and 429 HTTP status codes**.

# HTTP Status Codes: 503, 504, and 429

## 1. 503 Service Unavailable

### Meaning

`503 Service Unavailable` means the server is temporarily unable to process the request.

In simple words:

> The service is running but currently unavailable due to overload, maintenance, or dependency failure.

### When Does It Occur?

#### 1. Server Overload

The application receives too many requests and runs out of resources such as CPU, memory, threads, or connections.

Example:

* Thread pool exhausted
* High traffic spike

#### 2. Application Maintenance or Restart

The service may be temporarily unavailable due to:

* Deployment in progress
* Pod restart
* Server maintenance

Example:
A Kubernetes pod restarts and temporarily becomes unavailable.

#### 3. Dependency Failure

The application depends on external systems such as:

* Database
* Redis
* Kafka
* Third-party APIs

If those systems become unavailable, the service may return `503`.

Example:
A Spring Boot application cannot connect to the database.

#### 4. Connection Pool Exhaustion

If database connection pools are exhausted, requests cannot be processed.

Example:
All database connections are already occupied.

#### 5. Circuit Breaker Protection

In microservices, a circuit breaker may intentionally reject traffic to protect the system when downstream services are unhealthy.

### Example Response

```http
HTTP/1.1 503 Service Unavailable
```

### How to Resolve `503`

1. Check application logs for failures.
2. Verify CPU and memory usage.
3. Check dependency availability (database, cache, APIs).
4. Increase application replicas or resources.
5. Verify connection pool configuration.
6. Ensure Kubernetes readiness/liveness probes are healthy.

### Real-Life Analogy

> A restaurant is open, but the kitchen is overloaded or temporarily unavailable.

## 2. 504 Gateway Timeout

### Meaning

`504 Gateway Timeout` occurs when a gateway or proxy waits too long for a response from the backend service.

In simple words:

> The gateway asked the backend for a response, but the backend took too long.

### When Does It Occur?

#### 1. Slow Database Queries

A poorly optimized query takes too much time.

Example:

* Missing indexes
* Full table scans

#### 2. Slow Business Logic

Backend processing is slow.

Example:

* Large file generation
* Heavy report processing
* Processing millions of records

#### 3. Slow Downstream Service

In microservices:

Client → API Gateway → Service A → Service B

If Service B responds slowly, the request may timeout.

#### 4. Network Latency

Network communication problems between systems may delay responses.

Example:

* DNS delays
* Packet loss
* High latency

#### 5. Thread Starvation or Deadlocks

The application becomes slow because threads are blocked or fully occupied.

### Example Response

```http
HTTP/1.1 504 Gateway Timeout
```

### How to Resolve `504`

1. Identify slow operations using logs and monitoring.
2. Optimize database queries and indexing.
3. Reduce processing time of APIs.
4. Introduce async processing for long-running tasks.
5. Configure circuit breakers and timeouts.
6. Increase gateway timeout only when necessary.

### Real-Life Analogy

> A waiter goes to the kitchen for your food but waits too long and returns saying there is no response yet.

## 3. 429 Too Many Requests

### Meaning

`429 Too Many Requests` occurs when a client sends too many requests within a specific time period.

In simple words:

> You are sending requests too quickly. Slow down.

### When Does It Occur?

#### 1. API Rate Limiting

Public or private APIs restrict request volume.

Example:

* Maximum 100 requests per minute

If exceeded:

```http
HTTP/1.1 429 Too Many Requests
```

#### 2. Login Protection

To prevent brute-force attacks, systems throttle repeated login attempts.

Example:
Multiple failed login attempts in a short time.

#### 3. Gateway Throttling

API gateways may limit requests per:

* User
* API key
* IP address

Example:
100 requests/minute per IP.

#### 4. System Protection

The system intentionally slows traffic to avoid overload and preserve stability.

Instead of crashing and returning `503`, the service proactively limits traffic using `429`.

### Example Response

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 60
```

Meaning:

> Retry after 60 seconds.

### How to Resolve `429`

#### Client Side

1. Reduce request frequency.
2. Respect `Retry-After` header.
3. Implement retry with exponential backoff.
4. Batch requests when possible.

#### Server Side

1. Adjust rate limit configuration.
2. Add caching.
3. Scale infrastructure.
4. Improve request handling efficiency.

### Common Rate Limiting Algorithms

* Token Bucket
* Leaky Bucket
* Fixed Window Counter
* Sliding Window

### Real-Life Analogy

> A customer repeatedly presses a service bell too quickly, and the staff asks them to wait before requesting again.

## Difference Between 503, 504, and 429

| Status Code | Meaning                         | Main Cause                                | Who Usually Returns It          |
| ----------- | ------------------------------- | ----------------------------------------- | ------------------------------- |
| 503         | Service temporarily unavailable | Overload, maintenance, dependency failure | Backend service / load balancer |
| 504         | Gateway timeout                 | Backend response too slow                 | Gateway / reverse proxy         |
| 429         | Too many requests               | Client exceeded request limit             | API gateway / backend           |

### Quick Interview Summary

* **503 Service Unavailable** → Service is temporarily unable to process requests due to overload, maintenance, or dependency issues.

* **504 Gateway Timeout** → Gateway waited too long for backend response and timed out.

* **429 Too Many Requests** → Client exceeded configured rate limits and must retry later.

## 4. 501 Not Implemented

### Meaning

`501 Not Implemented` means the server understands the request but does not support or implement the functionality required to process it.

In simple words:

> The server understood the request, but this feature or request type is not supported.

### When Does It Occur?

#### 1. Unsupported HTTP Method

The server does not support the HTTP method used by the client.

Example:

Your API supports only `GET` and `POST`, but the client sends:

```http
PATCH /users/1
```

Response:

```http
HTTP/1.1 501 Not Implemented
```

Meaning:

> The server does not implement PATCH handling.

#### 2. Feature Not Implemented

A requested feature exists conceptually but has not been implemented.

Example:

A file management system supports file download but upload functionality is not implemented.

Client request:

```http
PUT /upload
```

Response:

```http
HTTP/1.1 501 Not Implemented
```

#### 3. Unsupported Protocol Capability

A proxy, server, or gateway receives a request capability or extension it does not support.

Example:

An older server receives a newer HTTP feature it cannot understand or process.

### Example Response

```http
HTTP/1.1 501 Not Implemented
```

### How to Resolve `501`

#### Client Side

1. Verify supported HTTP methods in API documentation.
2. Use a supported endpoint or request type.
3. Check API version compatibility.

#### Server Side

1. Implement the missing functionality.
2. Add support for the required HTTP method.
3. Upgrade server/framework if protocol support is missing.
4. Return `405 Method Not Allowed` instead if the method exists but is disallowed for that endpoint.

### 501 vs 405 (Important Interview Difference)

| Status Code | Meaning                                           | Example                         |
| ----------- | ------------------------------------------------- | ------------------------------- |
| 501         | Functionality/method is not implemented by server | Server does not support `PATCH` |
| 405         | Method exists but is not allowed on the endpoint  | `DELETE /users` not allowed     |

Example:

If a Spring Boot API has:

```java
@GetMapping("/users")
public List<User> getUsers() {
    return service.getUsers();
}
```

Request:

```http
DELETE /users
```

Typical response:

```http
HTTP/1.1 405 Method Not Allowed
```

Reason:

> Server understands DELETE, but `/users` endpoint does not allow it.

### Real-Life Analogy

> You go to a restaurant and ask for sushi, but the restaurant never prepares sushi at all.

(Compare this with `503`, where the restaurant normally serves sushi but is temporarily unavailable.)
